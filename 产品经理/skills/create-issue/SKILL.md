# 标准化 GitHub Issue 创建流程

> 所有通过 Claude Code 创建的 GitHub Issue 必须遵循此模板。
> 目标：让任何开发者看到 issue 后能**直接动手**，不需要再问"这个是什么意思"。

---

## Step 1: 信息收集

在创建 issue 前，先收集必要信息：

- [ ] 确认目标仓库（从 git remote 自动检测，或用户指定 `--repo`）
- [ ] 阅读相关代码，理解当前实现
- [ ] 确认问题的具体表现（错误信息、日志、截图）
- [ ] 确认影响范围和严重程度
- [ ] 搜索已有 issue，避免重复创建

---

## Step 2: 按模板组织内容

每个 issue **必须**包含以下 6 个部分：

### 模板结构

````markdown
## 背景

### 问题描述
<!-- 用 1-3 句话说清楚：什么坏了 / 缺什么 / 为什么要改 -->

### 错误示例
<!-- 具体的错误信息、截图、复现步骤，或当前代码的问题片段 -->
<!-- 如果是新功能，写"当前行为"vs"期望行为" -->

```python
# ❌ 当前代码 — 问题在哪
current_problematic_code()

# 或者错误日志
ERROR: xxx at line xxx
```

---

## 需求

### 实现逻辑
<!-- 描述要做什么，怎么做，分步骤说清楚 -->

1. 第一步：做什么
2. 第二步：做什么
3. 第三步：做什么

### 字段/接口变更
<!-- 如果涉及数据结构、API、配置项变更，用表格列出 -->

| 字段/接口 | 变更类型 | 说明 |
|----------|---------|------|
| `xxx` | 新增/修改/删除 | 具体说明 |

### 代码示例
<!-- 给出改后代码的示例（伪代码或实际代码均可） -->

```python
# ✅ 改后的代码应该长这样
improved_code()
```

---

## 验收标准

- [ ] 标准1：具体的、可验证的条件
- [ ] 标准2：具体的、可验证的条件
- [ ] 标准3：具体的、可验证的条件
<!-- 每条标准必须是"能明确判断 pass/fail"的，不要写模糊的 -->
<!-- 好的："`POST /api/xxx` 返回 200 且包含 `id` 字段" -->
<!-- 差的："API 能正常工作" -->

---

## 元信息

**优先级**: P0 (立即) / P1 (本周) / P2 (本月) / P3 (待定)

**相关文件**:
- `path/to/file1.py` — 说明
- `path/to/file2.jsx` — 说明

**关联 Issue**:
- #XX — 依赖/被依赖/相关
- #YY — 前置条件
````

---

## Step 3: 内容质量自检

创建前逐条检查：

### 背景部分
- [ ] 问题描述 ≤ 3 句话，任何人都能看懂
- [ ] 有具体的错误代码/日志/截图，不是泛泛而谈
- [ ] 说明了"为什么这是个问题"（影响是什么）

### 需求部分
- [ ] 实现逻辑有清晰的步骤，不是"请修复"
- [ ] 如果涉及数据变更，有字段/接口映射表
- [ ] 有改后代码示例（至少是伪代码）

### 验收标准
- [ ] 每条标准都是**可验证的**（能明确判断 pass/fail）
- [ ] 覆盖了正常路径和异常路径
- [ ] 不包含模糊表述（"正常工作"、"性能好"等）

### 元信息
- [ ] 优先级已标注且合理
- [ ] 列出了所有相关文件（不是让开发者自己找）
- [ ] 关联 issue 已标注依赖关系（依赖/被依赖/相关）

---

## Step 4: 确定优先级

| 级别 | 含义 | 判断标准 |
|------|------|----------|
| **P0** | 立即处理 | 生产环境故障、数据丢失、安全漏洞、阻塞其他人 |
| **P1** | 本周完成 | 功能缺失影响核心流程、严重体验问题 |
| **P2** | 本月完成 | 优化改进、非关键 bug、代码质量提升 |
| **P3** | 待定 | 锦上添花、长期规划、探索性需求 |

优先级前缀映射到 title 中的 emoji：
- P0 → 🔴
- P1 → 🟡
- P2 → 🟠
- P3 → 🔵

---

## Step 5: 展示给用户确认

**⚠️ 创建 issue 前，必须先展示完整内容给用户确认。**

展示格式：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 Issue 预览 — 请确认后创建
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📌 仓库: owner/repo
🏷️ 标签: bug / enhancement
🔴 优先级: P0

📝 标题: <标题>

📄 正文:
<完整的 issue 正文>

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
确认创建？(y/n)
```

**只有用户确认后才执行 `gh issue create`。**

---

## Step 6: 创建 Issue

```bash
gh issue create --repo <owner/repo> \
  --title "<emoji> <标题>" \
  --label "<bug|enhancement>" \
  --body "<正文>"
```

创建成功后输出 issue URL。

---

## ⚠️ 强制规则

1. **6 个部分缺一不可** — 背景、错误示例、需求、验收标准、优先级、相关文件
2. **验收标准必须可验证** — 不接受模糊表述
3. **必须有代码上下文** — 相关文件路径 + 错误/改进代码片段
4. **创建前必须用户确认** — 展示完整预览后再创建
5. **不创建重复 issue** — 先搜索已有 issue
6. **关联 issue 必须标注** — 如果有依赖或相关 issue

---

## 使用方式

```bash
# 基本用法
/create-issue 描述问题或需求

# 指定仓库和优先级
/create-issue 描述问题 --repo owner/repo --priority P1

# 批量创建（如 code review 后）
/create-issue 根据 review 结果创建 issues
```
