## 使用方式

每个项目维护一份架构决策文件。当架构方向发生重大变更时更新，用于：
- PR review 时判断代码是否和架构方向冲突
- 需求拆解时判断新 issue 是否符合当前架构
- 清理 open PR/Issue 时判断哪些过时

## 模板

```markdown
## 核心架构

（用文字或 ASCII 图描述系统的核心分层/通信方式）

## 关键决策

### 1. [决策标题]
- **决定**：xxx
- **原因**：xxx
- **影响**：xxx

### 2. [决策标题]
- ...

## 这些决策影响的 review 判断
- PR 做了 xxx → 和决策 1 冲突，关闭
- PR 做了 yyy → 需要确认是否符合决策 2
```

## 维护规则

- 架构变更时立即更新，不要事后补
- 记录"决定了什么"和"为什么"，不记录实现细节
- 每条决策附带"影响 review 判断"的说明——让 PM 在 review 时能快速判断
- 决策被推翻时标注"已废弃"+ 被哪条新决策取代，不要删除
