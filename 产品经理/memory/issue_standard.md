## 6 部分模板

每个 Issue 必须包含：

### 1. 背景
- 问题描述（≤3 句话，任何人能看懂）
- 错误示例（代码片段 / 日志 / 截图 / 当前行为 vs 期望行为）

### 2. 需求
- 实现逻辑（分步骤说清楚）
- 字段/接口变更表（用表格列出字段名、类型、变更类型）
- 改后代码示例（至少伪代码）

### 3. 验收标准
- 可验证的 checklist（能判断 pass/fail）
- 覆盖正常路径和异常路径
- 不接受模糊表述（"正常工作"、"性能好"）
- 好的例子："`POST /api/xxx` 返回 200 且包含 `id` 字段"
- 坏的例子："API 能正常工作"

### 4. 优先级
- P0 🔴 立即 — 生产故障、数据丢失、安全漏洞
- P1 🟡 本周 — 核心功能缺失、严重体验问题
- P2 🟠 本月 — 优化改进、非关键 bug
- P3 🔵 待定 — 锦上添花、长期规划

### 5. 相关文件
- 具体路径 + 说明（不让开发者自己找）

### 6. 关联 Issue
- 依赖 / 被依赖 / 相关 / 被取代

## 强制规则

- 6 个部分缺一不可
- 创建前必须展示给用户确认
- 先搜索已有 Issue，避免重复
